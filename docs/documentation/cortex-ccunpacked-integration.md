# HomeHub Cortex x ccunpacked.dev

Source reference:

- https://ccunpacked.dev/

## Goal

This integration does not copy the site visually.
It borrows the site's strongest logic:

- show the full request loop
- expose architecture as something explorable
- group capabilities by job instead of by file
- separate stable and experimental abilities

For HomeHub, this logic now lives inside the cortex blueprint itself.

## What changed in cortex

`runtime/cortex/architect.py` now emits four new explainability blocks:

- `requestLoop`
- `architectureExplorer`
- `capabilityExplorer`
- `featureStatus`

This means every brain generated by `AgentCortex.get_brain(...)` is now both:

- an execution blueprint
- an explanation blueprint

## Mapping

### 1. Agent Loop -> Request Loop

`ccunpacked.dev` shows:

- Input
- Message
- History
- System
- API
- Tokens
- Tools
- Loop
- Render
- Hooks
- Await

HomeHub cortex now exposes a simpler brain-native loop:

- `input`
- `normalize`
- `retrieve`
- `decide`
- `plan`
- `execute`
- `render`
- `learn`

This fits HomeHub better because HomeHub is multimodal and memory-driven.
It needs to show:

- OCR / voice / text intake
- memory lookup
- reuse vs create smart unit
- execution and packaging
- write-back learning

### 2. Architecture Explorer -> Cortex Explorer

`ccunpacked.dev` lets users inspect the source tree as a product surface.

HomeHub cortex now mirrors that idea with zones:

- `runtime`
- `cortex`
- `features`
- `memory`
- `services`
- `agent-blueprint`

This helps answer:

- where a request entered
- where it was routed
- where memory came from
- what smart unit is active

### 3. Tool System -> Capability Explorer

`ccunpacked.dev` groups tools by what they do.

HomeHub now groups capabilities by job:

- `conversation`
- `perception`
- `memory`
- `research`
- `execution`
- `adaptation`

This matches HomeHub more naturally than showing raw files or raw APIs.

### 4. Hidden Features -> Feature Status

`ccunpacked.dev` clearly marks hidden or experimental things.

HomeHub cortex now exposes statuses for:

- stable
- experimental
- planned

Current examples:

- `Voice Runtime`: stable
- `OCR Fallback Layers`: stable
- `Semantic Memory Learning Loop`: stable
- `Qdrant Vector Backend`: experimental
- `Autonomous Agent Creation`: experimental
- `Background Agents`: planned
- `LoRA Export Pipeline`: planned
- `Fine-Tuning Pipeline`: planned

## Why this matters

This changes HomeHub cortex from "brain only" to "brain plus observability".

That makes HomeHub better for:

- debugging
- user trust
- explaining why a request reused or created a smart unit
- showing which memory layer answered
- separating mature abilities from future ones

The most direct next step is to expose these new cortex fields in the dashboard:

- `requestLoop`
- `architectureExplorer`
- `capabilityExplorer`
- `featureStatus`

That would give HomeHub a true "Unpacked" mode instead of only keeping this logic inside the cortex blueprint.
